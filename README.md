Lobrecs Agent is a powerful developer desktop application built with Electron, React, and TypeScript. It serves as a local orchestrator and execution harness for multiple AI coding agents, dynamically routing tasks to the best-fit model based on task complexity, managing localized git-isolated workspaces, and offering a unified dashboard for swarm coordination, diff reviews, and cost auditing.

---

## 💡 Why Lobrecs Agent?

Standard agent command-line interfaces (CLIs) and browser extensions are usually tightly coupled to a single AI provider and lack system-level awareness of task complexity. They either overpay by running expensive frontier models (like Claude 3.7 Sonnet) for trivial tasks (such as fixing a typo or running a simple test), or they struggle to scale up when complex, multi-file architectural refactors are requested. 

Lobrecs Agent solves these limitations through several core design principles:

### 1. Complexity-Based Model Routing
Instead of hardcoding a single AI model, Lobrecs routes tasks dynamically.
* **Complexity Scorer (`src/main/router/ComplexityScorer.ts`)**: Evaluates prompt patterns, token size, reasoning keywords (e.g., *architect, database, schema, migration*), creation keywords (e.g., *implement, build*), cross-service signals (e.g., *preload, ipc, adapter*), and security risk signals.
* **Model Tiers**: Maps the complexity score to four model tiers:
  * `lightweight` (e.g., Gemini 2.0 Flash Lite for quick, simple fixes)
  * `balanced` (e.g., Gemini 2.5 Flash for standard functions and single-file modifications)
  * `advanced` (for multi-file features)
  * `frontier` (e.g., Claude 3.7 Sonnet via Claude Code for deep architectural and reasoning-heavy refactors)
* This dynamic routing significantly reduces API costs while ensuring the best reasoning engine is deployed when needed.

### 2. Multi-Agent Swarm Orchestration
Lobrecs is capable of orchestrating multiple specialized agents working in tandem. Rather than a single chatbot trying to plan, write, and review its own code, Lobrecs can execute agent pipelines (swarms) with dedicated roles (such as a programmer agent and a reviewer agent) to iteratively improve implementation quality.

### 3. Local Git-Isolated Workspaces (Worktrees)
To protect your main repository from corrupted states or uncommitted edits during execution, Lobrecs can isolate agent runs inside dedicated git worktrees. This guarantees that background agent trials do not interfere with your active working directory.

### 4. Interactive Monaco Diff Viewer & Auto-Application
* Diffs generated by coding agents are parsed and automatically applied in the main process to guarantee execution consistency.
* The frontend features a rich, side-by-side Monaco Editor interface for interactive code review, allowing the developer to audit changes before final integration.

### 5. Local Persistent History & Cost Tracking
* All agent activities, run logs, and execution inputs are streamed via a real-time terminal powered by `xterm.js` and `node-pty`.
* Token consumption, model usage, and pricing are persisted locally in a SQLite database (via `better-sqlite3`), providing a transparent dashboard of agent productivity and costs over time.

### 6. Project Memory (`.lobrecs/memory.json`)
* The agent automatically captures learned lessons, failures, and repository-specific architecture rules inside a repo-local `.lobrecs/memory.json` file.
* Future agent executions in the same repository automatically ingest this memory block, ensuring the agents learn and adapt to the project's codebase patterns over time.

## 🏗 Architecture & Under the Hood

Lobrecs Agent is structured around a strict process boundary to guarantee security, performance, and clear separations of concern.

```text
renderer modules ──> preload api ──> shared contracts <── main ipc ──> main application ──> domain
                                                                                   ──> infrastructure
```

### 1. Process Separation & Security Boundaries
* **Main Process (`src/main/`)**: Handles all privileged operations including filesystem writes, SQLite queries, child process execution (`node-pty`), git worktree management, global shortcuts, and update checks.
* **Preload Bridge (`src/preload/index.ts`)**: Exposes only narrow, typed APIs (`window.agentforge`) to the renderer. Renderer code never imports Node APIs, SQLite, `node-pty`, or filesystem helpers directly.
* **Renderer Process (`src/renderer/`)**: A Codex-style chat IDE built with React 19 and Tailwind CSS v4. Features a resizable 2-pane shell:
  * **Left Pane**: Collapsible project/session trees, thread history, and model/swarm configurations.
  * **Workspace View (Center)**: Session message timelines, working state progress rings, run command pills, and interactive diff views.
  * **Right Panel**: A togglable terminal drawer (`xterm.js`) and Monaco Diff Viewer.

### 2. Modular Feature Modules
The codebase is organized into process-aware feature modules. Each major feature (e.g., `agents`, `sessions`, `settings`, `swarms`, `updates`) owns its code across the application layers:
* `shared/contracts/<feature>.ts`: Holds serializable types, IPC payload definitions, and validation schemas that cross process boundaries.
* `main/modules/<feature>/domain/`: Pure domain rules and validation logic, 100% testable without Electron, SQLite, React, or filesystem access.
* `main/modules/<feature>/application/`: Coordinating use-cases and services.
* `main/modules/<feature>/infrastructure/`: Persistence repositories, CLI process runners, and external adapters.
* `main/modules/<feature>/ipc/`: Binds Electron IPC channels to validated service calls.

### 3. Agent Adapters (`src/main/agents/`)
Lobrecs wraps each external coding CLI using standardized adapters:
* **Claude Code (`ClaudeCodeAdapter.ts`)**: Spawns the `claude` CLI with `--output-format json`, allowing the main process to parse its stdout stream as JSONL objects containing tool calls, working states, and command executions.
* **Codex (`CodexAdapter.ts`)** and **OpenCode (`OpenCodeAdapter.ts`)**: Interfaces with Codex CLI and OpenCode runtimes to extract lists of available models and execute tasks in the terminal harness.
* **Antigravity (`AntigravityAdapter.ts`)**: An adapter designed to execute Gemini models directly using custom system prompts and tools.

## 🛠 Getting Started (For Contributors)

If you want to contribute to the Lobrecs Agent project, follow these instructions to set up the development environment.

### Prerequisites
* **Node.js** (v18 or higher recommended)
* **npm** (comes packaged with Node.js)
* **Agent CLIs**: Install the CLIs you wish to orchestrate (e.g., `@anthropic-ai/claude-code`, `codex`, `opencode`) globally on your system.

### Installation
Clone the repository and install the dependencies:
```bash
git clone https://github.com/oliveirabalsa/lobrecs-agent.git
cd lobrecs-agent
npm install
```

### Running Locally
To launch the Electron app in development mode with hot-reloading:
```bash
npm run dev
```

### Running Tests
Execute unit tests via Vitest:
```bash
npm test
```

### Type Checking & Production Build
To perform TypeScript compilation checks and bundle the application:
```bash
npm run build
```

## 🤝 Contribution Guidelines & Conventions

To keep the codebase clean, stable, and secure, all contributors must adhere to the following rules:

### 1. Local Command Execution Prefixing (`rtk`)
For local environment alignment and process auditing, all local shell commands executed within this repository (such as git commands, npm tasks, or test runs) must be prefixed with `rtk`.
* **Correct**: `rtk npm test`, `rtk git status`, `rtk npm run build`
* **Incorrect**: `npm test`, `git status`

### 2. Conventional Commits
Commit messages must follow the Conventional Commits specification:
* Format: `feat(scope): message` or `fix(scope): message`
* Example: `feat(updates): add manual download trigger`

### 3. Zero Credentials in Persistence
* **Never** hardcode or store API keys, session tokens, or private provider credentials in the source code, SQLite database, or browser local storage.
* Let the agent CLIs resolve credentials from their global environment paths or system keychain.

### 4. Pure Domains
* Domain rules and business logic must remain pure. Do not import React, Electron main/renderer APIs, SQLite, or filesystem operations inside any `domain/` folder.
* Keep these classes and utility functions easily testable with simple Vitest unit tests.

### 5. IPC Integrity
* Never expose raw database rows or file handles to the renderer process.
* Keep IPC channel names feature-scoped (e.g., `sessions:cancel-run`). Mirror and type all new IPC operations in `src/preload/index.ts` and `shared/contracts/`.
